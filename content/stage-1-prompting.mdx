## INSERT COIN — 왜 결과가 매번 별로였을까

Claude Code가 못해서가 아니다. **정보를 안 줬기 때문**이다.

오락실 게임도 컨티뉴 없이 1코인으로 깨려면 패턴을 알아야 한다.
프롬프트도 똑같다. 똑같은 모델한테 똑같은 일을 시켜도, **어떻게 시키느냐**에
따라 한 번에 끝나기도 하고 다섯 번을 다시 시켜야 하기도 한다.

---

## ROUND 1 — 나쁜 예 vs 좋은 예

### ❌ BAD — "로그인 기능 만들어줘"

```text
로그인 기능 만들어줘
```

무슨 일이 벌어지냐면:

- 어떤 인증 방식인지 모르니 **멋대로 가정**한다 (세션? JWT? OAuth?)
- 기존 코드 구조를 모르니 **새 패턴을 섞어** 들어온다
- 한 번에 코드를 다 쏟아내서, 방향이 틀리면 **전부 다시** 해야 한다

> 결과: "어… 이게 아닌데" → 다시 설명 → 또 다름 → 컨티뉴 코인 낭비

### ✅ GOOD — 제약과 범위를 먼저

```text
지금 인증은 NextAuth로 세션 기반이야. (app/api/auth/ 참고)
DB는 Prisma + Postgres고, User 모델엔 이미 email/passwordHash가 있어.

이메일+비밀번호 로그인을 추가하고 싶어.
코드는 아직 짜지 말고, 방향만 3가지 정도로 먼저 제안해줘.
각 방식의 트레이드오프도 한 줄씩.
```

무슨 일이 벌어지냐면:

- **기존 스택을 알려줬으니** 그 위에 자연스럽게 얹는다
- **"코드는 아직"** 이라고 했으니 방향부터 합의하고 간다
- 방향이 틀려도 **한 문장만** 고치면 된다

## ROUND 2 — 핵심 원칙 3가지

### 1. 제약 조건을 먼저 (CONSTRAINTS FIRST)

모델은 네 머릿속을 못 본다. 당연하게 여기는 것을 **명시**해라.

- 스택: `Next.js 14 / Prisma / Postgres`
- 하지 말 것: `새 라이브러리 추가 금지`, `기존 API 시그니처 유지`
- 스타일: `우리는 함수형 컴포넌트만 써`

> 제약이 없으면 모델은 "가장 흔한 방법"을 고른다. 그게 네 프로젝트랑 맞을
> 확률은 생각보다 낮다.

### 2. 범위를 명시 (DECLARE THE SCOPE)

"방향만", "이 파일만", "테스트는 빼고" — **경계를 그어줘라.**

```text
이 파일(src/lib/cart.ts)만 손대줘.
다른 파일은 건드리지 말고, 필요하면 뭘 바꿔야 하는지 말로만 알려줘.
```

범위를 안 주면 모델은 친절하게도 **여기저기 다 고친다.** 그리고 너는
diff 50개를 리뷰하게 된다.

### 3. 단계를 나눠라 (SPLIT INTO STAGES)

한 번에 다 시키지 마라. **방향 확인 → 코드 작성** 두 판으로 나눠라.

```text
[1단계] 방향만 제안해줘. 코드 금지.
        ↓ (내가 확인하고 고른 뒤)
[2단계] 좋아, 2번 방향으로 가자. 이제 코드 짜줘.
```

> 게임도 보스 패턴부터 보고 들어간다. 코드도 방향부터 보고 짜야 한 번에
> 깬다.

## BONUS STAGE — 바로 쓰는 템플릿

복붙해서 빈칸만 채워라.

```text
[컨텍스트] 지금 구조는 ___ 이고, 스택은 ___ 야.
[제약]     ___ 는 하지 마. ___ 는 유지해줘.
[목표]     ___ 를 하고 싶어.
[범위]     일단 ___ 만. 코드는 ___ (아직 / 짜도 됨).
[출력]     ___ 형태로 줘. (방향 3가지 / diff / 설명만)
```

---

## CLEAR! — 이번 판 요약

- 한 줄 명령 대신 **컨텍스트 · 제약 · 범위 · 단계**를 줘라
- 큰 작업일수록 **방향 확인 → 코드 작성**으로 쪼개라
- 모델이 "알아서" 한 것은 대개 **네 프로젝트와 다르다**

다음 판에서는 손이 키보드를 떠나지 않게 해주는 **단축키**를 배운다. ▶
